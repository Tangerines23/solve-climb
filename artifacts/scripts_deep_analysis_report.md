# 프로젝트 자동화 스크립트 심층 분석 및 보고

본 보고서는 프로젝트의 품질 및 성능 가드레일 역할을 수행하는 핵심 자동화 스크립트들에 대한 심층 분석 결과를 담고 있습니다. 단순히 실행 여부를 확인하는 것을 넘어, 실제 작동 원리와 프로젝트의 아키텍처적 부합성을 진단합니다.

---

## 1. JS Heap Snapshot Auditor (`check-heap-snapshot.js`)

### 작동 원리
- **기술 스택**: Playwright + Chrome DevTools Protocol (CDP).
- **메커니즘**:
  1. 실제 브라우저(Chromium)를 띄우고 애플리케이션에 접속합니다.
  2. `HeapProfiler.collectGarbage` 명령을 통해 강제로 가비지 컬렉션을 수행하여 'Baseline' 메모리를 측정합니다.
  3. 페이지 전환 및 리로드 시뮬레이션을 통해 런타임 부하를 줍니다.
  4. 다시 GC를 수행한 후 'Final' 메모리를 측정하여 그 차이(Growth)가 임계치(10MB)를 넘는지 확인합니다.

### 심층 분석
- **관리 범위**: 메모리 누수(Memory Leak) 가능성이 가장 높은 전역 상태(Zustand), 이벤트 리스너, 타임아웃 정리를 감시합니다.
- **프로젝트 부합성**: 퀴즈 게임 특성상 반복적인 렌더링과 복잡한 상태 업데이트가 발생하므로, 장기 세션에서의 안정성을 보장하는 핵심 도구입니다.
- **수정/개선점**: 현재는 단순 리로드만 수행하고 있으나, 향후 `Quiz` 페이지 진입/퇴장을 반복하는 구체적인 시나리오를 추가하여 컴포넌트 언마운트 시의 정리 로직을 더 정밀하게 검증해야 합니다.

## 2. Dependency Duplication Guardian (`check-dependency-duplication.js`)

### 작동 원리
- **기술 스택**: Node.js `child_process` + `npm ls`.
- **메커니즘**:
  1. `npm ls --all --json`을 통해 전체 의존성 트리를 JSON으로 추출합니다.
  2. 트리를 순회하며 각 패키지명에 대해 등록된 버전들을 수집합니다.
  3. 동일 패키지에 2개 이상의 버전이 존재할 경우 에러를 발생시킵니다 (화이트리스트 제외).

### 심층 분석
- **관리 범위**: 번들 크기 비대화(Bundle Bloat)를 방지하고, 서로 다른 버전의 라이브러리 충돌(예: 여러 버전의 React)로 인한 런타임 오류를 예방합니다.
- **프로젝트 부합성**: 수직 슬라이스 구조로 모듈화가 진행됨에 따라 의존성 관리가 복잡해질 수 있는데, 이를 빌드 전 단계에서 강제함으로써 청크 크기를 최소화합니다.
- **수정/개선점**: `react-router`나 `supabase` 등 크기가 큰 라이브러리의 중복은 즉시 차단되도록 임계치 기반 필터링을 추가할 수 있습니다.

## 3. Interaction Latency Auditor (`check-rerender-cycle.js`)

### 작동 원리
- **기술 스택**: Playwright + Web Vitals API.
- **메커니즘**:
  1. 브라우저에 `PerformanceObserver`를 주입하여 `event` 엔트리를 실시간 수집합니다.
  2. `processingEnd - startTime` 공식을 사용하여 사용자 클릭 후 다음 페인트까지의 지연 시간을 계산합니다.
  3. 100ms를 초과하는 상호작용이 발견되면 경고 또는 실패 처리합니다.

### 심층 분석
- **관리 범위**: 리액트의 불필요한 리렌더링 및 무거운 계산 로직으로 인한 사용자 경험 저하를 감시합니다.
- **프로젝트 부합성**: 퀴즈 정답 선택 시 즉각적인 피드백(애니메이션 등)이 중요한 앱이므로, 렌더링 병목을 잡는 데 매우 효과적입니다.
- **수정/개선점**: 현재 `startTime` 기반의 단순 측정이므로, 브라우저 스케줄링의 영향을 최소화하기 위해 여러 번 반복 측정 후 평균값을 사용하는 방식으로 고도화가 가능합니다.

## 4. 종합 평가 및 향후 제언

현재 프로젝트의 스크립트들은 **"예방적 방어(Preventive Defense)"** 전략에 최적화되어 있습니다. 단순히 코드가 돌아가는지를 보는 것이 아니라, **번들 크기, 메모리 안정성, 반응성**이라는 3대 핵심 품질을 CI/CD 파이프라인의 문지기로 세워둔 상태입니다.

### 추가 권장 사항 (Hardening Phase 5 제안)
1. **Network Stress Test**: 네트워크 지연이나 오프라인 환경에서의 PWA 작동성을 검증하는 스크립트 추가.
2. **Bundle Visualization**: 빌드 시마다 `visualizer` 리포트를 자동 생성하여 청크 간의 의존성 관계를 시각적으로 모니터링.
3. **Logic Fuzzing**: 퀴즈 생성 로직 등 복잡한 수식 계산부에 대해 무작위 입력을 주는 Fuzzing 테스트 강화.
