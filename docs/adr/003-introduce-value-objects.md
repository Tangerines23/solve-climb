# ADR 003: 값 객체(Value Object)와 스마트 생성자 도입

## 상태 (Status)
승인됨 (Accepted)

## 맥락 (Context)
현재 프로젝트는 도메인 데이터(콤보, 고도, 스태미나 등)가 원시 타입(`number`, `string`)으로 관리되는 '원시 타입 집착(Primitive Obsession)' 현상이 있습니다. 이로 인해 다음과 같은 문제점이 발생합니다:
1. **비즈니스 규칙 파편화**: 데이터의 유효성 검증이나 계산 로직이 여러 Hook이나 컴포넌트에 흩어져 있음 (예: 피버 레벨 계산, 콤보 최대치 제한 등).
2. **무결성 보장 어려움**: 유효하지 않은 데이터가 스토어에 저장되는 것을 방지하기 위해 매번 수동으로 체크해야 함.
3. **코드 가독성 저하**: 단순 숫자가 무엇을 의미하는지, 어떤 제약 사항이 있는지 코드를 깊이 파헤쳐야 알 수 있음.

## 결정 (Decision)
도메인 핵심 데이터에 **값 객체(Value Object)**와 **스마트 생성자(Smart Constructor)** 패턴을 도입합니다.

### 핵심 규칙:
1. **불변성 (Immutability)**: 모든 값 객체는 생성 후 상태가 변하지 않으며, 변경이 필요할 경우 항상 새로운 객체를 반환합니다.
2. **자기 검증 (Self-Validation)**: `private constructor`와 `static create()` 메서드를 사용하여 유효하지 않은 객체 생성을 원천 차단합니다.
3. **ROP (Rail-oriented Programming)**: `create()` 메서드는 에러를 던지지 않고 `{ success, data, error }` 형태의 Result 객체를 반환하여 안전한 파이프라인 구성을 지원합니다.
4. **비즈니스 로직 응집**: 해당 데이터와 관련된 모든 비즈니스 규칙(예: 고도 증가, 피버 단계 판별)은 값 객체 내부의 메서드로 캡슐화합니다.

## 결과 (Consequences)
- **도메인 무결성 향상**: 잘못된 도메인 데이터가 시스템 내부로 유입되는 것을 방지합니다.
- **테스트 용이성**: 순수 함수 형태의 값 객체 메서드들은 별도의 모킹 없이도 매우 쉽게 단위 테스트가 가능합니다.
- **보일러플레이트 증가**: Zustand 스토어의 `persist` 미들웨어 사용 시, 로컬 스토리지에서 데이터를 읽어올 때 클래스 인스턴스로 재구성하는 `merge` 로직이 추가로 필요합니다.
- **UI 독립성**: UI 컴포넌트는 값 객체 자체에 의존하지 않고, `.value` 프로퍼티를 통해 원시값만 추출하여 렌더링함으로써 결합도를 낮춥니다.
