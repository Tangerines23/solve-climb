# ADR 004: Auth 도메인 값 객체(Value Object) 및 스마트 생성자 도입

## 상태 (Status)
승인됨 (Accepted)

## 컨텍스트 (Context)
기존 Auth 도메인에서는 유저의 티어(`number | null`)와 이메일(`string | null`)을 원시 타입(Primitive)으로 관리하고 있었습니다. 이로 인해 다음과 같은 문제점이 발생했습니다:
1. **빈약한 도메인 모델 (Anemic Domain Model)**: 데이터에 대한 유효성 검증과 포맷팅 로직이 UI 컴포넌트나 훅에 흩어져 있었습니다.
2. **익명 유저 처리의 복잡성**: 이메일이나 티어가 없는 익명 유저 상태를 확인하기 위해 코드 곳곳에서 `null` 체크나 조건부 렌더링 로직이 반복되었습니다.
3. **타입 안전성 부족**: 원시 타입을 직접 사용함으로써 잘못된 범위의 값이나 형식이 주입될 위험이 있었습니다.

## 결정 (Decision)
Auth 도메인의 핵심 속성인 `Tier`와 `Email`을 **값 객체(Value Object)**로 전환하고 **스마트 생성자(Smart Constructor)** 및 **Null Object 패턴**을 도입하기로 결정했습니다.

### 1. Tier 값 객체
- **스마트 생성자**: `Tier.create(value: number | null)`를 통해 유효한 범위(0~6)를 검증합니다.
- **Null Object 패턴**: `null`이나 `undefined`가 입력될 경우 `Tier.UNRANKED` 인스턴스를 반환하여 UI에서의 깨짐을 방지합니다.
- **행위 캡슐화**: `getDisplayName()`, `getColor()`, `isRanked()` 등의 메서드를 통해 티어 관련 메타데이터를 객체 내부에서 관리합니다.

### 2. Email 값 객체
- **스마트 생성자**: `Email.create(value: string | null)`를 통해 정규식 기반의 엄격한 유효성 검증을 수행합니다.
- **마스킹 지원**: `getMasked()` 메서드를 제공하여 개인정보 보호 로직을 도메인 내부로 캡슐화합니다.
- **Null Object 패턴**: 익명 유저를 위한 `Email.ANONYMOUS` 인스턴스를 제공합니다.

### 3. 상태 저장소 통합
- Zustand `useProfileStore`의 상태 구조를 VO 기반으로 변경했습니다.
- `persist` 미들웨어의 `merge` 로직을 업데이트하여 로컬 스토리지에서 복원될 때 스마트 생성자를 통해 VO 인스턴스로 자동 변환(Hydration)되도록 구현했습니다.

## 결과 (Consequences)
1. **UI 코드 단순화**: UI 컴포넌트에서 복잡한 `if (tier === null)` 분기문이 사라지고 `tier.getDisplayName()` 등으로 직관적인 표현이 가능해졌습니다.
2. **응집도 향상**: 데이터 검증, 포맷팅, 비즈니스 룰이 도메인 객체 내부로 모여 유지보수가 용이해졌습니다.
3. **견고한 시스템**: 익명 유저 상태가 시스템 전체에서 일관되게 처리되며, 잘못된 형식의 데이터가 도메인 경계를 넘지 못하도록 차단되었습니다.
4. **테스트 용이성**: 복잡한 UI 렌더링 테스트 대신 도메인 로직에 대한 단순한 단위 테스트로 핵심 비즈니스 룰을 검증할 수 있게 되었습니다.
